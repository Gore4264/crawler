# ADR-0003: Один Postgres для всего в Phase 0

- **Дата**: 2026-04-25
- **Статус**: Accepted
- **Решение принял**: Делегировано продукт-агенту владельцем
- **Связанная точка решения**: D4 (из `ROADMAP.md` раздел 2)
- **Связанные ADR**: —

## Контекст

Архитектура `crawler` строится по принципу «Postgres-first»: один Postgres
держит сразу данные (mentions, signals, projects), эмбеддинги (через
pgvector), полнотекст (BM25 через GIN), очередь сообщений (через pgmq),
event bus (через LISTEN/NOTIFY) и distributed locks. Альтернатива (Kafka,
Redis, Elasticsearch отдельно) сознательно отвергнута в `ARCHITECTURE.md` —
лишние moving parts.

D4 спрашивает следующий уровень: жить ли всему этому в **одном инстансе**
Postgres, или физически разнести на два — `data-pg` (данные/индексы) и
`queue-pg` (pgmq + шина). Разделение даёт изоляцию workload-ов: высокий
churn по WAL от очереди не мешает аналитическим запросам по данным; разный
tuning под разные нагрузки.

## Опции

1. **Один Postgres-инстанс для всего.** Один docker-container, один backup,
   одна connection pool, один tuning-профиль.
2. **Два Postgres-инстанса** (`data-pg` + `queue-pg`). Два контейнера, два
   backup, два пула, разный tuning.

## Решение

**Один Postgres 16 с расширениями `pgvector`, `pgmq`, `plpgsql` для всех нужд
системы в Phase 0.**

Обоснование (решение принято продукт-агентом по делегации):

- **Реальная нагрузка Phase 0 на 2–3 порядка ниже** того, где разделение
  начинает окупаться. Phase 0 — это десятки-сотни событий **в час**, не
  в секунду; 3 источника со сканом раз в 5–60 минут. Один Postgres на
  скромном VPS тянет это с запасом 100–1000×.
- **Соответствует принципу владельца «сначала минимально работающая
  система, расширения по мере»** (зафиксирован при принятии D2). Один
  контейнер, один backup, одна точка наблюдения.
- **Постгрес-first** в архитектуре указывает ровно на этот выбор —
  максимальное использование одного инстанса, никаких лишних компонентов.
- **Разделение потом — недорогая миграция**: выделить queue-таблицы в
  отдельный namespace, поднять второй инстанс, перенести pgmq + триггер
  шины, обновить connection-config в `storage/`. Один день работы. До
  этого — лишняя сложность без выгоды.

## Последствия

**Становится возможным:**
- `docker-compose.yml` объявляет один service `postgres` с named volume.
- `storage/CLAUDE.md` (E1) — все репозитории, очереди и шина используют
  один connection pool через `asyncpg`/`SQLAlchemy`.
- Транзакции могут охватывать одновременно бизнес-данные и события шины
  (одна БД = настоящие ACID-границы между ними).

**Становится невозможным/дорогим:**
- Независимое масштабирование data vs queue нагрузки.
- Разный tuning Postgres под разные workload-ы.

**Что делать на следующих шагах:**
- В `storage/CLAUDE.md` (E1) — единая `Database` абстракция, единый pool.
- Все миграции (`storage/migrations/`) — против одной БД.
- В `bus/postgres_bus.py` (E4) — использует тот же pool.

**Триггеры для пересмотра (Phase 1+):**
- pgmq-таблицы становятся «hot» и VACUUM не успевает.
- LISTEN-сессии упираются в `max_connections`.
- Аналитические запросы по `signals` начинают занимать секунды из-за
  конкуренции с queue-нагрузкой.
- Любой из этих симптомов — открыть новый ADR на разделение, мигрировать
  queue в отдельный инстанс.

**Риск:** flood событий из streaming-источника (например, Bluesky firehose
без правильной фильтрации) может перегрузить single-Postgres. Митигация
заложена в архитектуре: BudgetGuard, scan-rate-limits, фильтрация на
уровне источника (всё в E4).
