# ADR-0002: Третий источник Phase 0 — Bluesky; Telegram MTProto отложен

- **Дата**: 2026-04-25
- **Статус**: Accepted
- **Решение принял**: Egor (владелец)
- **Связанная точка решения**: D2 + D3 (из `ROADMAP.md` раздел 2)
- **Связанные ADR**: —

## Контекст

Phase 0 покрывает 3 источника. Reddit (REST-pull, E1) и RSSHub (агрегатор
десятков платформ через RSS, E3) — определены. Третий нужен для **архитектурной
валидации** plugin-абстракции `BaseSource`: проверить, что она тянет паттерн
«long-lived streaming connection», принципиально другой от REST-pull.

Параллельно решается D3 — каким способом ходить в Telegram public, если он
выбран как этот третий источник: через MTProto/Telethon (нужен реальный
Telegram-аккаунт + SIM) или через Bot API (без аккаунта, но не умеет искать
по публичному Telegram, только по каналам где бот сидит).

Ключевое ограничение: владелец живёт во Вьетнаме, и личный Telegram-аккаунт —
критический канал коммуникации в реальной жизни (банки, доставка, expat-
сообщества). Risk-asymmetry использования личного аккаунта для крауллера
неприемлемая (потеря личного аккаунта при бане = катастрофа). Заведение
отдельного аккаунта на отдельной SIM возможно, но добавляет операционной
сложности и откладывает запуск системы.

## Опции

1. **Bluesky firehose через AT Protocol Jetstream**
   - WebSocket-консьюмер, public, без аккаунта/ключа.
   - Бесплатно. Python: `atproto` SDK или прямой WebSocket.
   - Аудитория: англоязычная tech/AI-тусовка.
   - Архитектурно: чистый streaming-паттерн.

2. **Telegram public через MTProto/Telethon на личном аккаунте**
   - Полное покрытие public Telegram (broad search).
   - Риски: бан личного аккаунта = потеря критического канала коммуникации.

3. **Telegram public через MTProto/Telethon на отдельном аккаунте + SIM**
   - Покрытие как у (2), без риска для личного аккаунта.
   - Цена: SIM ($1–10) + setup-время (1–2 часа) + откладывает запуск Phase 0.

4. **Telegram через Bot API**
   - Нет broad-search, только каналы где бот добавлен админом.
   - Не валидирует streaming-паттерн (REST-polling).

5. **Telegram через RSSHub `telegram/channel/{name}`**
   - Web-парсинг публичного превью канала.
   - Per-channel, последние ~20 сообщений, без поиска.
   - Архитектурно — REST-pull, не валидирует streaming.

## Решение

**D2: Bluesky firehose через AT Protocol Jetstream — третий источник Phase 0.**

**D3: отложен до Phase 1.** В Phase 0 Telegram через MTProto не подключаем.
Telegram-каналы при необходимости подключаются через RSSHub Telegram module
(REST-pull, без аккаунта) — это **четвёртый источник без D-решения**, потому
что архитектурно он не отличается от Reddit/других REST-pull источников.

**Принципиальное обоснование:** владелец явно зафиксировал приоритет —
сначала запустить минимально работающую систему end-to-end, источники
расширять по мере необходимости после запуска. Bluesky закрывает
архитектурную задачу D2 (валидация streaming-паттерна) с нулевой
операционной стоимостью и нулевым риском для личных активов.

## Последствия

**Становится возможным:**
- В E3 подключаем Bluesky firehose через `atproto` или прямой WebSocket
  к Jetstream. `BaseSource` валидируется на streaming-паттерне.
- При желании добавляем мониторинг конкретных Telegram-каналов через
  RSSHub в любой момент после E3 — без D-решения и без ADR.
- Никаких account-credentials или session-файлов в проекте → меньше
  attack surface, проще security model.

**Становится недоступным в Phase 0:**
- Broad search по Telegram (поиск по ключевым словам через всё public TG).
  Сценарии CONCEPT 3.4 (discovery людей в Дананге через Telegram) и любые
  сценарии где основной сигнал в Telegram-discussions — **не покрываются
  в Phase 0**.
- VN-coverage Telegram. Если такая потребность всплывёт раньше Phase 1 —
  возвращаемся к D3 с заведением отдельного аккаунта.

**Что делать на следующих шагах:**
- В E3 (`plugins/sources/CLAUDE.md` от архитектора) — задокументировать
  Bluesky как обязательный источник, с указанием на streaming-паттерн.
- В `plugins/sources/bluesky.py` (E3, ветка реализации) — WebSocket-консьюмер,
  фильтрация на клиенте, обработка disconnect/reconnect.
- В `BaseSource` контракте (закрепляется в `core/CLAUDE.md` в E0) —
  поддержать long-lived streaming через `AsyncIterator` + опционально
  `lifecycle hooks` (start/stop), чтобы streaming-источники могли держать
  соединение между scan-циклами. Архитектор уточнит это в E0.
- RSSHub Telegram channels — **не закладываем заранее**. Подключаются
  только если в Phase 0 окажется нужно мониторить конкретный TG-канал.

**Открытые на будущее (Phase 1+):**
- Когда (если) понадобится broad-search по Telegram — открыть новый ADR,
  сравнить актуальные на тот момент опции (отдельный аккаунт через
  Telethon vs TGStat API vs Apify-акторы) с учётом реальных потребностей.
- Условие триггера: владелец явно говорит «нужен поиск по TG» или новая
  стратегия проекта требует TG-discovery.

## Что это решение НЕ говорит

- Не говорит, что Telegram «не важен». Говорит, что в Phase 0 он подключается
  только в форме мониторинга известных каналов через RSSHub, без MTProto.
- Не отменяет возможность добавить MTProto в Phase 1 — структура plugin
  это позволяет, ADR будет тогда.
