# ADR-0004: Формат content_hash — только нормализованный текст

- **Дата**: 2026-04-25
- **Статус**: Accepted
- **Решение принял**: Делегировано продукт-агенту владельцем
- **Связанная точка решения**: D5 (из `ROADMAP.md` раздел 2)
- **Связанные ADR**: —

## Контекст

В таблице `mentions` стоит `UNIQUE (content_hash)` — техническая основа
дедупликации в pipeline. Стадия `DedupStage` считает хеш нового ментиона и
проверяет существование. Если коллизия — ментион отбрасывается, дальше pipeline
не идёт (экономия эмбеддингов и LLM).

D5 спрашивает: **что входит в `content_hash`** — только текст, или текст +
источник? Это точка невозврата на уровне SQL: схема создаётся в первой
миграции, поменять формат хеша потом = пересчитать весь корпус и переделать
UNIQUE constraint.

Реальный сценарий, на котором всплывает: одна и та же новость попадает на 5
платформ (Reddit + Bluesky + RSS + Telegram-репост + HN). Текст ±одинаковый,
источники разные.

## Опции

1. **`content_hash = sha256(normalized_text)`** — источник НЕ входит. 5 платформ
   = 1 запись (первый успевший побеждает), 1 запуск pipeline, 1 уведомление.
2. **`content_hash = sha256(normalized_text + source_id)`** — источник входит.
   5 платформ = 5 записей, 5 запусков pipeline (5× LLM-стоимость), 5
   уведомлений (если без второго слоя дедупа в notification stage).
3. **Гибрид**: текст-only хеш в `mentions` + отдельная таблица
   `mention_observations(mention_id, source, source_url, observed_at)` — один
   primary mention + отдельная запись per observation. Best of both, но
   сложнее схема.

## Решение

**`content_hash = sha256(normalized_text)`. Источник НЕ входит. Вариант 1.**

Решение принято продукт-агентом по делегации. Обоснование:

- **Cost-aware first-class** — фундаментальный принцип CONCEPT и ARCHITECTURE.
  Вариант 2 на каждом cross-source дубликате сжигает LLM-токены без выгоды.
  Вариант 1 запускает pipeline один раз на content.
- **«Шум — провал»** (CONCEPT). Вариант 2 без второго слоя дедупа в notifier
  отправит 5 уведомлений про одно событие.
- **Принцип владельца «минимально работающая система»** (зафиксирован 2026-04-25).
  Вариант 1 — простейший вариант: один UNIQUE constraint, никаких
  дополнительных таблиц или дедуп-слоёв.
- **Эволюция к варианту 3 — additive миграция**: можно добавить таблицу
  `mention_observations` отдельно, без перерасчёта хешей. Дверь к более
  богатой атрибуции остаётся открытой для Phase 1.

## Что входит и что НЕ входит в хеш

**Входит:** `normalized_text` после стадии Normalize в pipeline. Нормализация:
- Unicode NFKC.
- Lowercase.
- HTML-cleanup (вырезать теги).
- Collapse whitespace (свернуть последовательности пробелов/табов/переносов).
- (Опционально, на усмотрение архитектора в `core/CLAUDE.md`) — удалить
  URL-параметры трекинга (`utm_*` и т.п.) из inline-ссылок.

**Не входит:**
- `source_id` (по решению D5).
- `source_url` (один и тот же текст по разным URL — один контент).
- `author` (репост другим пользователем — один контент).
- `published_at` / `discovered_at` (один и тот же пост увиденный дважды — один
  контент).

**Алгоритм:** SHA-256 → hex-string → колонка `content_hash CHAR(64) NOT NULL UNIQUE`.

## Последствия

**Становится возможным:**
- Pipeline стадия `DedupStage` — простой SELECT по `content_hash`. O(log n)
  через UNIQUE-индекс.
- Cross-source дедупликация бесплатна: cross-platform реposts и квоты
  сворачиваются в одну запись автоматически.
- LLM-стадия запускается строго один раз per content.

**Становится недоступным/дорогим:**
- Атрибуция «эта новость прозвучала на N платформах» — недоступна без
  расширения схемы (вариант 3 как Phase 1+ миграция).
- Per-source метаданные сохраняются только для **первого** источника. Если
  Reddit отсканился раньше Bluesky — Bluesky-метаданные (URL, engagement)
  не сохранятся.
- Поведение «первый побеждает» означает: порядок сканов влияет на источник
  записи. Это арбитрарно, но не критично для KPI.

**Что делать на следующих шагах:**
- В `core/CLAUDE.md` (E0, архитектор) — финализировать формат normalize-pipeline
  с уровнем детализации до конкретных regex/правил, чтобы дедуп был
  детерминирован между разными запусками.
- В `storage/schema.sql` (E1) — `content_hash CHAR(64) NOT NULL UNIQUE`,
  индекс по `content_hash` (создастся автоматом из UNIQUE).
- В `processing/stages/dedup.py` (E1, ветка 2) — `SELECT 1 FROM mentions WHERE
  content_hash = $1` перед insert; при коллизии — drop с метрикой
  `dedup_dropped_total{reason="duplicate"}`.

**Триггер пересмотра (Phase 1+):**
- Появилась реальная задача анализа «социальный сигнал» (новость/тема
  прозвучала на N платформах) — добавить таблицу `mention_observations` через
  additive миграцию, начать писать observation-данные с момента миграции.
- Эта эволюция **не ломает** существующий хеш-формат и не требует пересчёта.
